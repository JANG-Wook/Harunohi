# 01. 제품 정의

## 1. 비전

> 비개발자도 시각적으로 챗봇을 만들고, 코드 몇 줄로 자사 웹/앱에 붙일 수 있는 임베디드 챗봇 SaaS.

차별화 가치 두 가지.

1. **"쉬움"** — 비개발자도, 엔터프라이즈 운영자도 빠르게 봇을 만들 수 있다.
2. **"하이브리드"** — 시나리오의 결정론적 안정성에 RAG의 자연스러움을 결합한다.

## 2. 사용자

| 사용자 | 역할 | 접점 |
|--------|------|------|
| **빌더 사용자** | 고객사의 운영자/기획자 — 봇 설계와 배포 | 우리 SaaS 콘솔 |
| **종단 사용자** | 고객사의 고객 — 챗봇으로 대화하는 사람 | 고객사 웹/앱에 임베드된 위젯 |

두 축이 분리된다는 점이 디자인, 운영, 요금 모델 전부에 영향을 준다.

## 3. 두 종류 시나리오

빌더가 다루는 시나리오는 두 종류로 명확히 구분한다.

| 구분 | FAQ 시나리오 | 기능 시나리오 |
|------|------------|--------------|
| **목적** | 질문에 답변 (CS 응대) | 업무 수행 (AS 신청, 예약 등) |
| **흐름** | 단발성 (질문 → 답변 → 종료) | 다단계 (변수 수집 → API → 분기) |
| **트리거** | 자유 발화 → 의도 추론 매칭 | 자유 발화 매칭 또는 메뉴 버튼 |
| **응답 방식** | 정적 답변 또는 RAG 응답 | 정적 메시지 + 입력폼 + API 응답 |
| **빌더 UI** | 간단한 Q&A 페어 + 문서 업로드 | 캔버스 (노드/에지) |

빌더 UI도 두 모드를 시각적으로 분리한다. FAQ를 캔버스에서 만들게 하면 비개발자가 떠난다.

## 4. 응답 생성 매트릭스

| 응답 모드 | 어디서 쓰임 | 처리 방식 |
|----------|-----------|-----------|
| 정적 메시지 | 양쪽 시나리오 모두 | 빌더에서 입력한 텍스트/카드 그대로 |
| RAG 응답 | 주로 FAQ | 업로드 문서 → 벡터 검색 → LLM이 문서 기반 자연어 응답 |
| API 응답 렌더링 | 기능 시나리오 | 외부 API 결과를 템플릿에 끼워 출력 (Phase 2) |
| 완전 자유 생성 | ❌ 제외 | 환각 위험 |

RAG는 캔버스의 **"RAG 응답 노드"** 형태로 제공한다. 노드 설정에서 참조할 지식베이스를 선택.

## 5. 의도 추론 — 토글 설정

봇 단위로 선택 가능.

| 옵션 | 장점 | 단점 |
|------|------|------|
| 임베딩 매칭 | 빠르고 저렴, 결정론적 | 학습 예시 다수 필요 |
| LLM 분류 | 학습 예시 적어도 동작 | 비용·지연·환각 위험 |

## 6. 핵심 모듈

### A. Authoring (저작) — MVP 필수
- 워크스페이스/봇 CRUD
- FAQ 모드 (질문/답변 페어 + 문서 업로드)
- 기능 시나리오 모드 (캔버스 노드/에지)
- 메시지 컴포넌트 라이브러리 (HailMary 자산 재활용)
- 변수 관리

### B. Simulation (시뮬레이션) — MVP 필수
- 빌더 옆 패널에서 즉시 대화 테스트
- 현재 노드/변수 상태 디버깅 표시

### C. Deployment (배포) — MVP 필수
- 버전 관리 (저장 ≠ 배포 분리)
- 발행/롤백
- 임베드 스니펫 발급

### D. Runtime (런타임) — MVP 필수
- 임베드 위젯 (플로팅 버튼 + 챗 윈도우)
- 대화 엔진 (세션, 상태머신, 변수 바인딩)
- 의도 추론 (임베딩 / LLM 토글)
- RAG 파이프라인

### E. Integration (외부 API 연동) — Phase 2
- API 호출 노드 (REST 우선)
- 요청/응답 변수 매핑
- 인증 설정 (Header / API Key / OAuth)

### F. Operations (운영) — Phase 3
- 대화 로그 검색
- 분석 대시보드
- 답변 품질 피드백

## 7. MVP 스코프

### 포함
1. 빌더 콘솔 — FAQ 모드, 기능 시나리오 모드
2. 메시지 컴포넌트 (HailMary 자산 통합)
3. 시뮬레이터
4. RAG 파이프라인 (문서 업로드 → 임베딩 → 벡터 검색)
5. 의도 추론 (임베딩 / LLM 토글)
6. 임베드 위젯
7. 버전 관리·배포

### 제외 (Phase 2 이후)
- 외부 API 연동 노드
- 분석 대시보드
- 협업 (멀티 유저 동시 편집)
- 엔터프라이즈 배포 자동화
- 멀티 채널 (카카오톡, 슬랙 등)

## 8. 배포 모델

| 항목 | SaaS (셀프 서비스) | Enterprise (구축형) |
|------|---------------------|---------------------|
| 배포 | 우리 클라우드 멀티테넌트 | 고객사 인프라 또는 VPC 격리 |
| 데이터 | 우리가 보관 | 고객사 보관 |
| LLM | 우리 API 키 | 고객사 키 또는 온프레 LLM |
| 가격 | 월 구독 (봇 수/대화수) | 라이선스 + 구축비 + 유지보수 |
| 진입 시점 | MVP부터 | Phase 2 이후 |

엔터프라이즈는 동일 코드베이스에서 배포 모드만 다르게 가져갈 수 있도록 처음부터 설계한다.

## 9. "쉬움"을 위한 구체 장치

- **템플릿 갤러리** — "AS 신청 봇", "예약 봇" 등 클릭 한 번에 시작
- **자연어 빌더 보조** — "이름과 휴대폰을 받는 단계를 만들어줘" → 노드 자동 생성 (LLM 활용)
- **인스펙터 단순화** — 노드 클릭 시 필수 필드만 노출, 고급은 접기
- **실시간 시뮬레이터** — 저장 없이도 즉시 테스트
- **검증 도구** — 미연결 노드, 미정의 변수를 빌더에서 표시
