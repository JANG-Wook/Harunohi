# 04. 의사결정 기록

## 확정된 결정

### D1. 봇 동작 방식 — 하이브리드 (시나리오 + RAG)
- **결정** — 시나리오 기반이 기본, RAG 응답은 노드로 제공
- **근거** — 결정론적 안정성(시나리오) + 자연스러움(RAG) 둘 다 필요
- **결정일** — 2026-05-20

### D2. 의도 추론 모드 — 봇 단위 토글 (임베딩 / LLM)
- **결정** — 봇 설정에서 둘 중 하나 선택 가능
- **근거** — 학습 예시가 풍부하면 임베딩, 적으면 LLM이 유리
- **결정일** — 2026-05-20

### D3. 답변 생성 범위 — 정적 + RAG, 자유 생성 제외
- **결정** — 정적 메시지와 RAG 응답까지. 완전 자유 LLM 생성은 제외
- **근거** — 환각 위험. 엔터프라이즈에서 받아들이기 어려움
- **결정일** — 2026-05-20

### D4. 타겟 — SMB부터 엔터프라이즈까지
- **결정** — 누구나 쓸 수 있는 빌더가 1차 목표. 엔터프라이즈는 추가 구축비
- **근거** — 인포뱅크 팀의 엔터프라이즈 경험 + "쉬운 빌더" 니즈
- **결정일** — 2026-05-20

### D5. 두 배포 모델 — SaaS + Enterprise
- **결정** — SaaS 멀티테넌트가 기본. 엔터프라이즈는 별도 구축
- **근거** — 보안 요구가 큰 엔터프라이즈는 인스턴스 분리가 필요
- **결정일** — 2026-05-20

### D6. 시나리오 두 종류 — FAQ + 기능
- **결정** — 시나리오 타입을 명시적으로 분리. 빌더 UI도 모드 분리
- **근거** — FAQ는 표/리스트, 기능은 캔버스가 자연스러움
- **결정일** — 2026-05-20

### D7. 기술 스택 — infobank-kakao-open-builder 기반
- **결정** — Spring Boot 3.5 / Java 21 / React 19 / Python FastAPI / MySQL
- **근거** — 검증된 코드, 팀 보유 역량, 패턴 재활용
- **결정일** — 2026-05-20

### D8. 세션/메시지 보관 기간 — 1년
- **결정** — 1년 보관, `created_at` 인덱스 + 일 1회 cleanup cron
- **근거** — 운영자가 충분히 돌아볼 수 있는 기간. 운영 데이터 보고 조정
- **결정일** — 2026-05-20

### D9. Vector DB — MySQL과 분리
- **결정** — Qdrant (또는 동급 OSS Vector DB) 별도 운영. MySQL에는 메타데이터만
- **근거** — MySQL은 벡터 검색을 네이티브 지원하지 않음
- **결정일** — 2026-05-20

### D10. 외부 노출 ID — ULID (`public_id`)
- **결정** — auto-increment PK는 내부용, 외부에는 ULID 노출
- **근거** — auto-increment 추측 공격 방지, ULID는 정렬 가능
- **결정일** — 2026-05-20
